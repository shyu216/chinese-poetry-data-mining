# 重构方案 - 2026-03-13

**项目**: chinese-poetry-data-mining  
**版本**: v1.0

---

## 一、问题诊断

### 1.1 数据管线问题

| 问题 | 现状 | 影响 |
|------|------|------|
| 数据散落 | 原始数据、中间结果、最终输出混在一起 | 难以追踪数据来源 |
| Schema不统一 | 诗/词/曲使用不同字段名 | 合并时需要手动映射 |
| 版本缺失 | 无版本号标识 | 无法追溯数据迭代 |
| Override机制缺失 | 各脚本缓存逻辑不一致 | 无法控制是否重新生成 |

### 1.2 可视化框架问题

| 问题 | 现状 | 影响 |
|------|------|------|
| 脚本冲突 | 04_build_index.py 和 08_export_github_pages.py 职责重叠 | 输出文件混乱 |
| 职责不清 | 可视化生成、索引构建、API服务混杂 | 维护困难 |
| CI/CD不完整 | 可视化生成被注释掉 | 无法自动更新 |

## 二、架构决策

### 2.1 数据分层架构

```
RAW (原始) → BRONZE (清洗) → SILVER (结构化) → GOLD (分析) → OUTPUT (输出)
```

| 层级 | 职责 | 输出格式 |
|------|------|----------|
| RAW | 原始数据，只读 | JSON (submodule) |
| BRONZE | 清洗、合并、繁简转换 | CSV + JSON 元数据 |
| SILVER | 结构化处理、格律提取 | CSV |
| GOLD | 分析结果（相似度、词汇频率） | CSV + JSON |
| OUTPUT | Web/API 输出 | JSON 分片 |

### 2.2 目录结构

```
data/
├── raw/                    # 原始数据 (只读)
├── bronze/                 # 清洗后数据
├── silver/                 # 结构化数据
├── gold/                   # 分析结果
├── output/                 # 最终输出
└── cache/                  # 临时缓存 (gitignore)
```

### 2.3 分析模块设计

| 分析器 | 方法 | 用途 |
|--------|------|------|
| TextSimilarityAnalyzer | TF-IDF + Cosine Similarity | 基于用词习惯发现风格相近的诗人 |
| WordFrequencyAnalyzer | jieba 分词 + 词性筛选 | 统计每位作者的高频实词 |
| MeterAnalyzer | 字数统计 + 模式匹配 | 格律分析 |
| POSTaggerAnalyzer | jieba.posseg | 词性标注和统计 |

**已移除的功能**:
- ❌ 社交网络分析（诗词中人名出现概率极低）
- ❌ 情感分析（诗词抽象，准确率低）

### 2.4 可视化模块设计

| 类型 | 模块 | 特点 |
|------|------|------|
| 纯图表 | 情感分布、朝代分布、格律统计 | 静态，无交互 |
| 交互应用 | 格律大全、作者相似度搜索 | 搜索、分页、筛选 |

### 2.5 API 设计

| 版本 | 端点 | 功能 |
|------|------|------|
| v1 | /poems, /authors, /stats | 基础查询 |
| v2 | /similarity, /words, /pos | 相似度、词汇、词性 |

## 三、性能与用户体验

### 3.1 渲染优化

| 问题 | 解决方案 |
|------|----------|
| 渲染卡顿 | 渐进式加载 + 虚拟滚动（最多100个DOM节点） |
| 图表数据过大 | 自动采样（散点图≤5000点，网络图≤500节点） |
| 功能未完成 | FeatureStatus 类统一管理状态，显示"开发中"提示 |

### 3.2 GitHub 大文件处理

| 限制 | 策略 |
|------|------|
| 单文件 ≤ 100MB | 每片 ≤ 10MB，约 10,000 首诗 |
| 仓库 ≤ 1GB | 中间数据不上传，只上传分片后数据 |

## 四、扩展性设计

### 4.1 插件化架构

```
Analyzer (基类)
  ├── TextSimilarityAnalyzer  ✅
  ├── WordFrequencyAnalyzer   ✅
  ├── MeterAnalyzer           ✅
  └── StyleEmbeddingAnalyzer  🚧 (未来)
```

### 4.2 扩展原则

1. 新增分析器 → 继承 BaseAnalyzer → 自动注册
2. 新增可视化 → 继承 BaseVisualizer → 组合使用
3. 新增 API → 添加到对应版本蓝图
4. 无需修改现有代码，符合开闭原则

## 六、关键决策

| 决策 | 理由 |
|------|------|
| 不向后兼容 | 旧代码 Schema 混乱，无法渐进式迁移 |
| 使用 CSV + JSON | 比 Pickle 更易读、易调试 |
| Pydantic Schema | 强类型验证，IDE 友好 |
| 分层架构 | 清晰的数据流向，便于追踪和调试 |
| 移除情感分析 | 诗词抽象，准确率低 |
| 移除社交网络 | 诗词中人名出现概率极低 |

## 七、代码规范

### 7.1 命名规范

| 类型 | 规范 | 示例 |
|------|------|------|
| 目录 | 小写+下划线 | `data/bronze/` |
| 文件 | 小写+下划线 | `01_clean.py` |
| 类 | 大驼峰 | `TextSimilarityAnalyzer` |
| 函数 | 小写+下划线 | `load_data()` |
| 数据文件 | 版本前缀 | `v1_poems_merged.csv` |

### 7.2 Git 忽略策略

- 大文件不上传（*.pkl, *.gz）
- 中间数据不上传（bronze/, silver/, gold/）
- 只上传分片后的 output/

## 九、总结

| 问题 | 解决方案 |
|------|----------|
| 数据散落 | 分层目录 (bronze/silver/gold/output) |
| Schema不统一 | Pydantic 强制类型定义 |
| Override机制缺失 | PipelineConfig.force 参数 |
| 脚本冲突 | viz/ 和 export/ 职责分离 |
| CI/CD混乱 | 统一 pipeline.py + 细化触发条件 |
| 可视化卡顿 | 渐进式加载 + 虚拟滚动 + 采样 |
| GitHub大文件 | 分片导出 + manifest + 懒加载 |
| 扩展性差 | 插件化架构 + 版本化API + 组件化可视化 |
